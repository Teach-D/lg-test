전체 서비스 품질 검증을 수행한다.

## 입력

$ARGUMENTS

## 실행 흐름

### 1단계: 프롬프트 분석 (prompt 스킬)

입력을 아래 6개 차원으로 분석한다.
- 목적(Goal), 맥락(Context), 출력 형식(Format), 톤/스타일(Tone), 성공 기준(Criteria), 예시(Examples)

이미 명확한 항목은 건너뛴다. 부족한 항목이 있으면 최대 3회까지 질문하여 보완한다.

검증 범위가 지정되지 않으면 git diff 기반으로 변경된 코드를 대상으로 한다.
"전체"를 요청하면 모든 앱을 검증한다.

### 2단계: 품질 검증 (qa 에이전트)

qa 에이전트에 작업을 위임한다. 에이전트는 아래 순서로 검증한다:

1. **범위 파악**: git diff로 변경 파일 확인, 영향받는 앱/도메인 판단
2. **정적 분석**: 타입 체크, 린트, 빌드 확인 (영향받는 앱만)
   - backend: `./gradlew ktlintCheck`, `./gradlew compileKotlin`
   - web-admin: `npx tsc --noEmit`, `npx eslint src/`
   - web-user: `npx tsc --noEmit`, `npx eslint src/`
   - app: `flutter analyze`
3. **테스트 실행**: 기존 테스트 실행 + 누락 영역 식별
4. **API 계약 검증**: 백엔드 DTO ↔ 프론트 타입 일치 확인
5. **보안 체크**: 인증 누락, 입력 검증, 시크릿 노출, XSS

### 3단계: 리포트 출력

아래 형식으로 결과를 출력한다:

```
## QA 리포트 — YYYY-MM-DD

### 요약
- 🔴 Critical: N건
- 🟡 Warning: N건
- 🔵 Info: N건
- 정적 분석: ✅/❌ (앱별)
- 테스트: ✅ N/N 통과 / ❌ N건 실패

### 발견 사항
#### 🔴 [BUG-001] 제목
- 심각도 / 위치 / 현상 / 예상 동작 / 재현 방법 / 수정 제안

### 테스트 누락 영역
- [ ] 누락된 테스트 케이스

### API 계약 불일치
- [ ] 불일치 필드

### 수동 검증 체크리스트
- [ ] 핵심 시나리오
```

코드를 수정하지 않는다. 발견한 문제에 대해 수정 제안만 포함한다.
