# Prompt Improver

사용자의 프롬프트를 분석하고, 질문을 통해 부족한 부분을 보완하여 개선된 프롬프트를 출력한다.

## 워크플로우

1. 사용자로부터 프롬프트를 받는다
2. 6개 차원으로 분석한다
3. 부족한 항목만 질문한다 (최대 3회)
4. 개선된 프롬프트를 마크다운 코드블록으로 출력한다
5. 사용자가 확인하면 해당 프롬프트로 작업을 시작한다

## 분석 차원

프롬프트를 아래 6개 차원으로 평가한다.

| 차원 | 판단 기준 |
|---|---|
| 목적(Goal) | 달성하려는 결과가 구체적인가 |
| 맥락(Context) | 배경, 대상, 상황, 제약조건이 충분한가 |
| 출력 형식(Format) | 결과물의 길이, 구조, 형태가 명시되었는가 |
| 톤/스타일(Tone) | 어조, 문체, 난이도가 특정되었는가 |
| 성공 기준(Criteria) | 좋은 결과를 판단할 기준이 있는가 |
| 예시(Examples) | 참고 예시가 필요하고 제공되었는가 |

## 분석 규칙

- 이미 명확한 항목은 질문하지 않는다 (✅ 표시)
- 프롬프트의 맥락에서 불필요한 차원은 건너뛴다
- 예시 차원은 복잡한 출력이 필요한 경우에만 평가한다

## 질문 규칙

- **최대 3회**까지만 질문한다
- 한 턴에 관련 질문을 2~3개 묶어서 질문 횟수를 절약한다
- 가능하면 선택지를 제공하여 답변 부담을 줄인다
- 가장 임팩트가 큰 부족 항목부터 우선 질문한다
- 부족한 항목이 없으면 질문 없이 바로 개선 프롬프트를 출력한다

## 출력 형식

### 1단계: 분석 결과

```
## 프롬프트 분석
- ✅ 목적: [명확한 이유]
- ⚠️ 맥락: [부족한 내용]
- ⚠️ 출력 형식: [부족한 내용]
- ✅ 톤: [명확한 이유]
- ⚠️ 성공 기준: [부족한 내용]
- ✅ 예시: [불필요 또는 제공됨]
```

분석 후 부족한 항목에 대해 질문한다.

### 2단계: 개선된 프롬프트

모든 질문이 끝나면 (또는 질문이 불필요하면) 아래 형식으로 출력한다.

````
## 개선된 프롬프트

```
[개선된 프롬프트 전문]
```

### 변경점
- [원본 대비 추가/수정된 내용 1]
- [원본 대비 추가/수정된 내용 2]
````

### 3단계: 확인 후 실행

사용자에게 확인을 요청한다. 사용자가 확인하면 개선된 프롬프트의 내용대로 작업을 시작한다.

## 예시

**입력:** "블로그 글 써줘"

**분석:**
```
## 프롬프트 분석
- ⚠️ 목적: 주제가 지정되지 않음
- ⚠️ 맥락: 대상 독자, 게시 플랫폼 불명확
- ⚠️ 출력 형식: 길이, 구조 미지정
- ⚠️ 톤: 문체 미지정
- ⚠️ 성공 기준: 없음
- ✅ 예시: 불필요
```

**질문 1회차:**
> 몇 가지 여쭤볼게요.
> 1. 어떤 주제의 블로그 글인가요?
> 2. 대상 독자는 누구인가요? (예: 개발자, 일반인, 마케터)
> 3. 톤은 어떤 걸 원하시나요? (1) 격식체 (2) 캐주얼 (3) 기술 문서 스타일

**답변:** "React 상태관리 비교, 주니어 개발자 대상, 캐주얼"

**질문 2회차:**
> 1. 분량은 어느 정도가 좋을까요? (1) 짧게 1000자 (2) 보통 3000자 (3) 길게 5000자+
> 2. 특별히 비교하고 싶은 라이브러리가 있나요?

**답변:** "3000자, Redux vs Zustand vs Jotai"

**개선된 프롬프트:**
````
## 개선된 프롬프트

```
React 상태관리 라이브러리 비교 블로그 글을 작성해줘.

- 주제: Redux vs Zustand vs Jotai 비교
- 대상: 주니어 프론트엔드 개발자
- 톤: 캐주얼하고 친근한 문체
- 분량: 약 3000자
- 구조: 도입 → 각 라이브러리 소개 → 비교표 → 결론(추천)
- 각 라이브러리마다 간단한 코드 예제 포함
```

### 변경점
- 비교 대상 라이브러리 3개 명시
- 대상 독자 "주니어 프론트엔드 개발자" 지정
- 톤 "캐주얼하고 친근한 문체" 지정
- 분량 "약 3000자" 지정
- 구조와 코드 예제 포함 요구사항 추가
````
